# 建联邮箱私有网红统计功能开发文档

## 一、功能概述

本功能用于统计每个网红顾问建联邮箱下的私有网红数据，支持多维度统计分析，帮助运营人员了解各建联邮箱的网红资源分布情况。

### 1.1 权限与数据范围

| 角色 | 数据范围 | 查询条件 |
|------|----------|----------|
| **部门经理/管理员** | 可查看全部网红顾问数据 | 1. 不选顾问、不选邮箱：查看全部顾问汇总统计<br>2. 选择顾问、不选邮箱：查看该顾问所有邮箱汇总统计<br>3. 选择顾问+邮箱：查看该顾问该邮箱的统计 |
| **网红顾问小组长** | 可查看自己及组内人员数据 | 1. 不选顾问、不选邮箱：查看自己+组内人员汇总统计<br>2. 选择组内人员、不选邮箱：查看该组员所有邮箱汇总统计<br>3. 选择组内人员+邮箱：查看该组员该邮箱的统计 |
| **网红顾问** | 仅可查看自己的数据 | 1. 不选邮箱：查看自己所有邮箱汇总统计<br>2. 选择邮箱：查看自己该邮箱的统计 |

### 1.2 查询逻辑流程图

```
部门经理/管理员访问：
┌─────────────────────────────────────────────────────────────┐
│  是否选择顾问？                                               │
│     ├── 否 → 展示全部顾问列表统计（汇总页面）                   │
│     └── 是 → 是否选择建联邮箱？                                │
│              ├── 否 → 展示该顾问所有邮箱汇总统计               │
│              └── 是 → 展示该顾问该邮箱的详细统计               │
└─────────────────────────────────────────────────────────────┘

网红顾问小组长访问：
┌─────────────────────────────────────────────────────────────┐
│  获取当前用户及组内人员ID列表                                  │
│  是否选择顾问？                                               │
│     ├── 否 → 展示自己+组内人员汇总统计（汇总页面）              │
│     └── 是（仅能选自己或组内人员）→ 是否选择建联邮箱？          │
│              ├── 否 → 展示该人员所有邮箱汇总统计               │
│              └── 是 → 展示该人员该邮箱的详细统计               │
└─────────────────────────────────────────────────────────────┘

网红顾问访问：
┌─────────────────────────────────────────────────────────────┐
│  自动获取当前登录用户ID作为顾问ID                              │
│  是否选择建联邮箱？                                           │
│     ├── 否 → 展示自己所有邮箱汇总统计                         │
│     └── 是 → 展示自己该邮箱的详细统计                         │
└─────────────────────────────────────────────────────────────┘
```

## 二、数据模型分析

### 2.1 涉及表结构

| 表名 | 说明 | 主要字段 |
|------|------|----------|
| `store_user_contact_email` | 网红顾问建联邮箱表 | id, sys_user_id, contact_email, email_status |
| `store_celebrity_private_counselor` | 私有网红网红顾问签约表 | celebrity_private_id, celebrity_contact_email, contract_time |
| `store_celebrity_private` | 私有网红表 | id, gender, platform_type, country_name, followers_num |
| `store_celebrity_private_attribute_relation` | 私有网红属性关联表 | celebrity_private_id, attribute_type_name, attribute_name |

### 2.2 表关联关系

```
store_user_contact_email (建联邮箱)
    │
    │ contact_email = celebrity_contact_email
    ▼
store_celebrity_private_counselor (签约表)
    │
    │ celebrity_private_id = id
    ▼
store_celebrity_private (私有网红表)
    │
    │ celebrity_private_id = celebrity_private_id
    ▼
store_celebrity_private_attribute_relation (社媒属性表)
```

## 三、统计维度设计

| 维度名称 | 字段来源 | 字段说明 | 统计分类 |
|----------|----------|----------|----------|
| 平台统计 | `platform_type` | 平台类型 | 0=Instagram, 1=YouTube, 2=TikTok |
| 性别统计 | `gender` | 性别 | 0=男, 1=女, null=未知 |
| 国家统计 | `country_name` | 所在国家 | 按国家名称分组 |
| 粉丝量级统计 | `followers_num` | 粉丝数 | 分区间统计：1K-10K、10K-20K、20K-50K、50K-100K、100K+ |
| 段位统计 | `is_top_star` | 段位等级 | 1=头牌, 2=A级, 3=黑名单, 0=普通 |
| 社媒属性统计 | `attribute_type_name` + `attribute_name` | 属性类型和属性名称 | 动态属性类型 |
| 签约时间统计 | `contract_time` | 签约时间 | 按年/月分组 |

## 四、接口设计

### 4.0 统一权限说明

所有统计接口需根据当前登录用户的角色自动控制数据范围：

```java
// 后端伪代码示例
public Result getStatistics(String sysUserId, String contactEmailId) {
    // 获取当前登录用户
    LoginUser loginUser = getCurrentUser();

    // 判断角色
    boolean isManager = hasRole(loginUser, "manager") || hasRole(loginUser, "admin");
    boolean isGroupLeader = hasRole(loginUser, "group_leader");

    if (isManager) {
        // 部门经理/管理员：可查看全部或指定顾问数据
        // sysUserId为空则返回全部，有值则返回指定顾问
    } else if (isGroupLeader) {
        // 网红顾问小组长：可查看自己和组内人员数据
        // 获取组内人员ID列表
        List<String> groupMemberIds = getGroupMemberIds(loginUser.getId());

        if (StringUtils.isEmpty(sysUserId)) {
            // 不选顾问：返回自己+组内人员汇总统计
            sysUserId = null; // 查询时使用组内人员ID列表
        } else {
            // 选择了顾问：校验是否为自己或组内人员
            if (!sysUserId.equals(loginUser.getId()) && !groupMemberIds.contains(sysUserId)) {
                throw new BusinessException("无权查看该顾问数据");
            }
            // 返回该顾问的统计
        }
    } else {
        // 网红顾问：只能查看自己的数据
        sysUserId = loginUser.getId();
    }

    // contactEmailId为空返回汇总，有值返回详细
    ...
}
```

**角色枚举定义：**

| 角色编码 | 角色名称 | 数据范围 |
|----------|----------|----------|
| `admin` | 管理员 | 全部数据 |
| `manager` | 部门经理 | 全部数据 |
| `group_leader` | 网红顾问小组长 | 自己+组内人员 |
| `counselor` | 网红顾问 | 仅自己 |

### 4.1 统计汇总接口

**接口路径：** `GET /celebrity/private/statistics/summary`

**功能说明：**
- 部门经理/管理员：不传sysUserId返回全部顾问汇总，传sysUserId返回该顾问汇总
- 网红顾问小组长：不传sysUserId返回小组汇总，传组内人员ID返回该人员汇总
- 网红顾问：自动使用当前用户ID，无需传sysUserId
- contactEmailId为空返回该顾问所有邮箱汇总，有值返回指定邮箱详情

**请求参数：**

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| sysUserId | String | 否 | 网红顾问ID（部门经理/管理员/小组长可传，网红顾问自动使用当前用户） |
| contactEmailId | String | 否 | 建联邮箱ID，不传则统计该顾问所有邮箱汇总 |
| startDate | String | 否 | 签约开始时间（格式：yyyy-MM-dd） |
| endDate | String | 否 | 签约结束时间（格式：yyyy-MM-dd） |

```sql
-- 统计某个建联邮箱下的私有网红各维度数据
SELECT
    e.id AS contact_email_id,
    e.contact_email,
    e.sys_user_name,
    COUNT(DISTINCT cp.id) AS total_celebrities,
    -- 平台统计
    SUM(CASE WHEN cp.platform_type = 0 THEN 1 ELSE 0 END) AS instagram_count,
    SUM(CASE WHEN cp.platform_type = 1 THEN 1 ELSE 0 END) AS youtube_count,
    SUM(CASE WHEN cp.platform_type = 2 THEN 1 ELSE 0 END) AS tiktok_count,
    -- 性别统计
    SUM(CASE WHEN cp.gender = 0 THEN 1 ELSE 0 END) AS male_count,
    SUM(CASE WHEN cp.gender = 1 THEN 1 ELSE 0 END) AS female_count,
    SUM(CASE WHEN cp.gender IS NULL THEN 1 ELSE 0 END) AS unknown_gender_count,
    -- 段位统计
    SUM(CASE WHEN cp.is_top_star = 1 THEN 1 ELSE 0 END) AS top_star_count,
    SUM(CASE WHEN cp.is_top_star = 2 THEN 1 ELSE 0 END) AS a_level_count,
    SUM(CASE WHEN cp.is_top_star = 3 THEN 1 ELSE 0 END) AS blacklist_count,
    SUM(CASE WHEN cp.is_top_star = 0 OR cp.is_top_star IS NULL THEN 1 ELSE 0 END) AS normal_count,
    -- 签约费用
    SUM(cpc.contract_amount) AS total_contract_amount,
    AVG(cp.followers_num) AS avg_followers
FROM store_user_contact_email e
LEFT JOIN store_celebrity_private_counselor cpc ON e.contact_email = cpc.celebrity_contact_email
LEFT JOIN store_celebrity_private cp ON cpc.celebrity_private_id = cp.id
WHERE e.email_status = 1
GROUP BY e.id, e.contact_email, e.sys_user_name;
```

## 七、前端界面设计建议

### 7.0 权限与界面控制

| 角色 | 默认展示 | 筛选条件 | 操作权限 |
|------|----------|----------|----------|
| **部门经理/管理员** | 全部顾问汇总统计（含平台、性别等分布图） | 网红顾问下拉框（可搜索，全部顾问）、建联邮箱下拉框（联动）、时间范围 | 可查看任意顾问数据 |
| **网红顾问小组长** | 小组汇总统计（含平台、性别等分布图） | 网红顾问下拉框（自己+组内人员）、建联邮箱下拉框（联动）、时间范围 | 只能查看自己及组内人员数据 |
| **网红顾问** | 自己所有邮箱汇总统计 | 建联邮箱下拉框（自己的邮箱）、时间范围 | 只能查看自己数据 |

**界面展示规则：**
- **部门经理/管理员不选择网红顾问时**：展示全部顾问的汇总统计数据，包含平台分布饼图、性别分布饼图、国家分布图、粉丝量级分布图、段位分布图、签约时间趋势图
- **部门经理/管理员选择某个网红顾问时**：展示该顾问的建联邮箱列表及各邮箱的网红数量，以及该顾问的各维度统计图
- **网红顾问小组长不选择网红顾问时**：展示自己+组内人员的汇总统计数据，包含各维度统计图
- **网红顾问小组长选择组内人员时**：展示该组员的建联邮箱列表及各邮箱的网红数量，以及该组员的各维度统计图
- **网红顾问**：始终展示自己的建联邮箱列表及各维度统计图

**概览卡片展示规则：**
- **默认展示5个指标**：建联邮箱数、总网红数、本周新增数、头牌数量、A级数量
- **选择了具体建联邮箱后**：隐藏"建联邮箱数"，仅展示4个指标（总网红数、本周新增数、头牌数量、A级数量）

## 八、注意事项

1. **数据量考虑**: 如果数据量较大，建议添加分页查询和缓存机制
2. **权限控制**:
   - 后端必须校验当前用户角色
   - **管理员/部门经理**：可查询全部或指定顾问数据
   - **网红顾问小组长**：可查询自己及组内人员数据，需校验sysUserId是否在组内
   - **网红顾问**：只能查询自己的数据（sysUserId强制使用当前用户ID）
   - 前端根据角色显示/隐藏筛选项（如顾问选择下拉框）
   - 小组长选择顾问时，下拉框仅显示自己+组内人员
3. **性能优化**: 对于复杂统计SQL，建议添加适当的索引
4. **数据一致性**: 统计数据可考虑定时任务预计算，存储到统计结果表中
5. **数据联动**:
   - 选择顾问后，建联邮箱下拉框应只显示该顾问的邮箱
   - 清空顾问选择后，建联邮箱下拉框也应清空
6. **小组长数据获取**:
   - 需要通过组织架构或用户关联关系获取组内人员ID列表
   - 建议在用户表或组织架构表中维护小组长与组员的关联关系

## 九、扩展功能建议

1. **趋势对比**: 支持不同时间段的数据对比
2. **导出功能**: 支持导出统计报表到Excel
3. **数据钻取**: 支持点击图表查看详细数据列表
4. **自定义维度**: 支持用户自定义统计维度和区间
5. **订阅报告**: 支持定期发送统计报告到邮箱
